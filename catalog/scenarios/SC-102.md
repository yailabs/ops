# SC-102 — Core Qualification: Contain → Evidence (No Mind) — Cross-Domain

## 1) Purpose
SC-102 qualifies the deterministic **core runtime** by proving that a single runtime grammar:

**trigger → context → authority/contract → decision → enforcement → evidence**

remains invariant across the nine D-Major domains (D1–D9). Only **Domain Pack semantics** change; the runtime does not.

SC-102 is the v0.1 credibility “Gate A”: it proves the infrastructure can **decide**, **contain**, and **produce evidence** without Mind and without UI.

## 2) Scenario model (parameterized)
SC-102 is parameterized by:

- **Domain Pack**: `Dk/<pack-id>` (one of D1..D9)
- **Workload**: a real target executed inside a workspace (or a reference workload)
- **Forbidden effect**: an external action not authorized by the pack semantics
- **Baseline contracts**: versioned baselines that determine `allow | deny | quarantine` (and optional `rate_limit`)

Same shape, different semantics.

## 3) Preconditions (runtime readiness)
The following must be true for a SC-102 execution:

- Runtime pipeline is active: **boot → root → kernel → engine** (no mind required).
- Workspace isolation is active and `ws_id` is non-forgeable.
- Control plane is active (session/context + role/arming where applicable).
- Engine is reachable through the mediated control path (root/kernel mediated mode is acceptable).
- Baseline contract/policy material is loaded and hashable.
- Decision vocabulary is stable: `allow | deny | quarantine | rate_limit` (where applicable).
- Evidence pipeline is active (trace stream + persistent materialization).

## 4) Trigger (stimulus)
The trigger is always:

**“a workload inside a workspace attempts a forbidden effect.”**

Domain examples (semantics differ; grammar is identical):
- **D1 Digital**: egress to a non-contracted destination / provider call outside registry.
- **D2 Physical**: actuator command outside authority envelope (via adapter/simulator).
- **D3 Biological**: access/export of sample data outside consent or chain-of-custody constraints.
- **D4 Social/Institutional**: procedural act executed without the required authorization step/role.
- **D5 Economic**: payment/credit authorization outside policy limits or contractual scope.
- **D6 Operational**: reroute/recovery action outside escalation/arming rules or time windows.
- **D7 Cognitive/Cultural**: publication/enforcement action without provenance or outside policy.
- **D8 Scientific**: experiment/pipeline run with invalid parameter lock or undeclared exclusions.
- **D9 Environmental/Climatological**: telemetry ingest or alert publication without calibration/provenance/parameter lock.

Important: the runtime enforces effects; it does not “understand philosophy.” Semantics are encoded by the pack.

## 5) Expected core behavior (grammar invariant)
For each Domain Pack execution, the runtime must perform:

### 5.1 Context
Emit and bind a context envelope including (minimum):
- `ws_id`
- `trace_id` (or equivalent correlation id)
- principal / role (identity/authority context)
- baseline hash (policy/contract hash)
- arming state (when relevant)

### 5.2 Authority / Contract
Resolve the pack’s contract/policy baseline (versioned and hashable) and apply authority constraints.

### 5.3 Decision
Produce an explicit decision:
- `outcome` (allow/deny/quarantine/...)
- `reason_code` (stable under the same baseline and stimulus)

### 5.4 Enforcement
Contain the forbidden effect:
- forbidden effect MUST NOT occur (fail-closed)
- enforcement must be attributable to the decision (not “best effort”)

### 5.5 Evidence
Produce structured evidence sufficient for replay/forensics:
- decision record with baseline hash and reason code
- evidence index or pointers
- metrics/KPIs required by the pack

## 6) Claims (what SC-102 proves)
C1. **Fail-closed containment**: non-authorized effects do not occur.  
C2. **Deterministic decision vocabulary**: outcome + reason code are stable under the same baseline and stimulus.  
C3. **Evidence-first**: each decision produces structured evidence anchored to baseline hashes.  
C4. **Workspace scoping**: evidence is unambiguous; `ws_id` cannot be forged.  
C5. **Cross-domain coherence**: changing packs changes semantics, not the runtime grammar.  
C6. **Minimal disclosure**: evidence includes only what is required (no unnecessary leakage).  

## 7) Non-goals
- Mind/LLM reasoning, automatic remediation, patch generation.
- UI/cockpit.
- Full enterprise orchestration replacement (ticketing/SOAR).

## 8) KPIs (minimum)
- **time-to-contain** (p50/p95)
- **forbidden effect success rate** (target: 0)
- **evidence completeness** (target: 100% required fields)
- **repeatability** (target: 3/3 consistent runs under same baseline)
- **cross-domain coverage** (number of D-Major domains proven with valid packs)

## 9) Acceptance (PASS / FAIL)
PASS if all are true:
- forbidden effect is blocked (fail-closed)
- decision record includes: outcome + reason_code + baseline hash
- evidence pack is complete and indexed
- 3 consistent runs for the same pack + baseline

FAIL if any are true:
- forbidden effect occurs even once
- evidence is missing, ambiguous, or untraceable
- a mandatory check is skipped (`SKIP`) in a way that impacts closure
